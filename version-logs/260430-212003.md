### 1. FRD-063a — Name Generation Controls & Edit-Mode Name Editing (Verified + Bug Fix)

**Status:** ✅ Implemented & verified

**What was already working:**
- `GeneratorDashboard` has descriptor mode UI (`clean` / `descriptive` / `verbose`)
- `App.tsx` pre-fills body name annotations into `localStorage` on generation
- `SystemViewer` system name input routes through `pendingSystem` in edit mode
- `SystemViewer` body names route through `pendingAnnotations` in edit mode
- Legacy systems auto-populate body names from `placeNames.bodyNames` on view

**Bug fixed:**
- `handleGenerateNames` in `SystemViewer.tsx` previously called `onUpdateSystem` directly even when in edit mode, bypassing the staging mechanism and saving immediately.

### 3. FRD-069h — Save As Copy Logic UX Polish

**Status:** ✅ Implemented

**Changes:**

1. **`useSystemEditMode.ts`**:
   - `saveAsNew()` now returns `Promise<StarSystem | undefined>` so callers can access the new system's ID
   - Clears `batchId: undefined` on copy — prevents the copied system from claiming a batch membership it doesn't actually have
   - Clears `diceLocks: undefined` on copy — locks are per-edit-session, not per-system
   - Default name suffix changed from `"(edited)"` to `"(copy)"` for clarity
   - Falls back to `"(copy)"` if user enters an empty prompt

2. **`SystemViewer.tsx`**:
   - `saveAsNew` callback now uses the **returned copied system's ID** when saving pending annotations to `localStorage`
   - Fixes a bug where annotations were saved to the **old** system's localStorage key, causing custom body names/notes to be lost on the copy
