- **Symptom**: Every new feature adds more conditional rendering, state, and callbacks to an already massive file
- **Impact**: Build times increasing (chunk now ~1.9MB), cognitive load for maintenance, risk of merge conflicts

- **Symptom**: `Cool`, `FrostLine`, `O1–O5` all get rank `5` (worst), worse than `Infernal`
- **Impact**: Outer-zone worlds lose tiebreakers unfairly
- **Location**: `src/lib/habitabilityPipeline.ts` lines 78–88
- **Fix complexity**: Low — add cases for v2 `ZoneId` values

### Problem 3: No component extraction for SystemViewer tabs
- **Symptom**: All tabs (OverviewTab, StarTab, WorldTab, InhabitantsTab) are nested functions inside SystemViewer.tsx
- **Impact**: Cannot lazy-load, cannot unit test in isolation, props drilling through 5+ levels

- **Symptom**: Economics dials don't auto-recalculate starport
- **Impact**: User must remember to click "Recalculate Starport & Trade" after changing dials
- **Fix complexity**: Medium — need to debounce recalculation and handle async ships-in-area generation

- **Symptom**: 30+ modified files, mix of user work and agent changes
- **Impact**: Risk of losing work, difficult to rollback, no clean commit history
- **Current state**: `git status` shows many uncommitted changes

## Root Cause Analysis (RCA)

### RCA 1: SystemViewer.tsx bloat
**Root cause**: No component boundaries were established when the project started. Every tab was implemented as an inline function rather than a separate module. Feature additions followed the path of least resistance (add more lines to the existing file).

**Prevention**: Extract each tab into `src/components/tabs/` directory. Use compound component pattern or context for shared state. Establish a "no file over 500 lines" rule.

**Fix**: Refactor InhabitantsTab into `src/components/tabs/InhabitantsTab.tsx` first (highest complexity), then OverviewTab, then StarTab/WorldTab.

### RCA 2: v2 ZoneId tiebreaker bug
**Root cause**: `habitabilityPipeline.ts` was written when only legacy `Zone` type existed (`Infernal`, `Hot`, `Conservative`, `Cold`, `Outer`). When FR-042 introduced `ZoneId` with `Cool`, `FrostLine`, `O1–O5`, the tiebreaker function was never updated.

**Prevention**: When extending enums/types, grep for all `switch` statements on that type and audit for missing cases. TypeScript's `strict` mode + `noFallthroughCasesInSwitch` catches some but not `default:` fallbacks.

**Fix**: Add explicit cases for `Cool`, `FrostLine`, `O1`–`O5` in `getZonePreferenceRank()`.

### RCA 3: Nested tab functions
**Root cause**: Same as RCA 1 — early architectural decision to colocate tabs with parent. React's prop-drilling pattern was used instead of context or state management.

**Prevention**: Use React Context or Zustand for SystemViewer-level state (system, edit mode, annotations). Each tab becomes a pure component receiving only what it needs.

### RCA 4: Manual starport recalculation
**Root cause**: Auto-recalculation requires importing `getGdpPerDayForWorld` and `calculateStarport` into the UI layer, which crosses the generator↔UI boundary. We avoided this to keep concerns separated.

**Prevention**: Extract a `recalculateEconomics(system, overrides)` pure function in `src/lib/economicsEngine.ts` that both generator and UI can call.

**Fix**: Create `src/lib/economicsEngine.ts` with `recalculateStarportFromDials()` helper. Call it on every dial change with `useEffect` + debounce.

### RCA 5: Uncommitted changes
**Root cause**: The user and agent are both modifying files without committing. No git workflow was established for agent sessions.

**Prevention**: Agent should propose a commit after each significant feature group. User should review and approve commits.

**Fix**: Run `git diff --stat` to categorize changes. Commit FRD-068/069/QA-070 as "feat: editable worlds v2 — raw udp, edit mode, starport floors". Commit R3 separately as "feat: population-scaled government terminology". Commit System Targets refactor separately.

## Decisions Made

1. **FRD-070 stays on hold** — Economic Classification requires population calibration (QA-067) and composition pipeline (FR-041) which are not yet verified. Implementing it now would create incorrect classifications.

2. **FRD-069 implemented as foundation only** — Edit mode has state management, economics dials, and save/discard. Add/Delete/Reposition worlds deferred to next sprint due to complexity (HSR checks, zone slotting, mainworld re-evaluation).

3. **Starport floors use habitatType + biochem heuristics** — Since FRD-070 economic classifications don't exist yet, R4 floors detect "Mining" via `habitatType.includes('Mining')` and "Agricultural" via `envHab > 0 && biochemResources ∈ {Abundant, Inexhaustible}`. This is a temporary heuristic until FRD-070 provides proper economic roles.

4. **RAW UDP profile cached on StarSystem** — `rawUdpProfile` is computed at generation time and stored. This avoids re-rolling the government code on every view toggle. The `mnemeSource.governmentRoll` field preserves the original roll for debug.

5. **Edit mode uses local pendingSystem, not Dexie** — Changes are buffered in React state until Save is clicked. This prevents half-edited systems from persisting and allows clean Discard.
