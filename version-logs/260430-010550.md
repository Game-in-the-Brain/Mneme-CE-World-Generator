## Root Cause Analysis (RCA)

**Blank body names when `includeNames` is ON**
- `App.tsx` generated `placeNames.bodyNames` but never copied them to `localStorage` annotations
- `PlanetarySystemBodies.tsx` reads from `annotations[body.id].name`, not `placeNames.bodyNames`
- Fix: copy `bodyNames` into annotations during generation and on SystemViewer mount

**Descriptor overload ("Commercial Bilaraltamira Hallowed Organized Bazpueblo...")**
- `@gi7b/placegen` `PlaceDescriptorEngine` rolls `1d3-1` descriptors (0–2) with equal probability
- With many bodies per system, ~1/3 of names get 2 descriptors, creating long 3-word names
- Fix: added `nameDescriptorMode` option; default `'descriptive'` caps at 1 descriptor, making 3-word names impossible

**Names not editable in edit mode**
- System name input used `system.name` directly, bypassing `pendingSystem`
- Body names saved immediately to `localStorage` via `handleAnnotation`, not staged
- Fix: system name now routes through `pendingSystem`; body names use `pendingAnnotations` staging

## Decisions Made

- **Default descriptor mode = `descriptive`**: Users found 3-word names overwhelming. Single-descriptor names feel more natural for most worlds.
- **Self-contained section components for Settings**: EconomicAssumptionsSection and LifeAssumptionsSection manage their own localStorage state. This reduces prop-drilling and makes the main Settings.tsx a thin shell.
- **Pending annotations pattern**: Similar to `pendingSystem`, `pendingAnnotations` stages body name/note changes during edit mode and commits on Save. This keeps the edit-mode mental model consistent.
