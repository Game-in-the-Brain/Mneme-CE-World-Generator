# Feature Requirements Document (FRD)
## 2D Animated Planetary System Map — MWG Integrated Build

**Date:** 2026-04-15  
**Status:** Spec Ready — Awaiting Implementation  
**Feature Code:** FR-031  
**Integration Model:** Monorepo sub-directory (NOT a fork or submodule)

---

## 1. Objective

Deliver a **one-tap visual star-system map** inside the Mneme CE World Generator (MWG). When a referee generates a `StarSystem`, a **"View System Map"** button opens a responsive, animated 2D canvas map of that system. All visual data is derived from the already-generated MWG state. The 2D map does **not** generate new astronomical data; it is a pure visualiser.

**Core user story:**
> *"I press a button, and I see my generated system as a live 2D map. I can watch planets orbit, step time forward or backward, zoom and pan, and share the exact same view with another player by sending them a short seed code."*

---

## 2. Core Design Principles

1. **MWG First, 2D Second** — All astronomical data (stars, zones, bodies, distances) is generated by MWG before the map ever renders. The 2D map is a read-only visualiser.
2. **Everything Is a Circle** — Abstraction over photorealism. Stars, planets, disks, and trajectory arcs are all geometric primitives.
3. **Seeded Procedural Visuals** — The starfield background is procedurally generated from a per-system seed. Two users with the same seed see the same background.
4. **One-Tap Launch** — A single button in MWG opens the map. No file exports, no manual JSON uploads.
5. **Phone-First Performance** — The renderer must run smoothly on mid-range phones and tablets.

---

## 3. Architecture

### 3.1 Monorepo Layout

The 2D map lives as a **second Vite entry point** inside the MWG repo.

```
Mneme-CE-World-Generator/
├── index.html                    ← Main React app
├── src/                          ← React + MWG engine
├── solar-system-2d/
│   ├── index.html                ← 2D map entry point
│   ├── src/
│   │   ├── main.ts               ← Bootstrap
│   │   ├── app.ts                ← State + event wiring
│   │   ├── renderer.ts           ← requestAnimationFrame loop
│   │   ├── camera.ts             ← Pan, zoom, transform math
│   │   ├── orbitMath.ts          ← Ellipse / angle / epoch helpers
│   │   ├── starfield.ts          ← Procedural background noise
│   │   ├── dataAdapter.ts        ← MWG StarSystem → scene graph
│   │   ├── uiControls.ts         ← DOM controls (play, pause, speed, seed)
│   │   └── styles.css
│   └── public/
│       └── (none — everything is vector/procedural)
└── vite.config.ts                ← Multi-page config
```

### 3.2 Why Not a Submodule?

Because the 2D map must iterate in lockstep with MWG data structures. A submodule creates two-repo friction (two commits, two pushes, version skew). Keeping it in one repo means:
- `npm run dev` serves **both** apps.
- Type changes in `src/types/index.ts` instantly propagate to `solar-system-2d/src/`.
- One GitHub Pages deployment hosts both.

## 4. Data Flow — The "Path"

### 4.1 Generation Order (Strict)

```
[User presses Generate in MWG]
           ↓
[MWG engine creates StarSystem JSON]
           ↓
[2D Map button becomes active]
           ↓
[User presses "View System Map"]
           ↓
[MWG serialises StarSystem + StarfieldSeed + EpochState]
           ↓
[MWG navigates to /solar-system-2d/?system=<base64>]
           ↓
[2D map deserialises, builds scene graph, starts render loop]
```

### 4.2 What Travels Across the "Path"

The map receives a **compressed payload** (Base64 JSON in the query string) containing:

```typescript
interface MapPayload {
  // The full star system already generated by MWG
  starSystem: StarSystem;

  // Procedural background seed (alphanumeric, e.g., "X7Kp9mQ2")
  starfieldSeed: string;

  // Epoch state
  epoch: {
    year: number;   // default 2300
    month: number;  // default 1
    day: number;    // default 1
  };

  // Optional: if user copied a seed from another system
  overrideStarfieldSeed?: string;
}
```

**Why Base64 in the query string?**
- No server required (GitHub Pages compatible).
- Shareable link: a referee can paste the URL to another player and they see the exact same system and background.
- Size is acceptable because MWG `StarSystem` JSON is ~3–5 KB uncompressed.

### 4.3 The "View System Map" Button

**Location:** Inside the main `SystemViewer` component, next to the existing **Export JSON** and **Export CSV** buttons.

**States:**
- **Disabled:** No system has been generated yet.
- **Active:** A system exists. Pressing it opens `/solar-system-2d/?system=<payload>` in the **same tab** (immersive mode) or a **new tab** (side-by-side with MWG).

**MVP decision:** Open in a **new tab** so referees can keep tweaking the system in MWG while referencing the map.

## 5. Visual Design

### 5.1 Body Rendering Rules

All bodies are filled circles with a 1px stroke.

| MWG Type | Visual | Fill | Stroke | Label |
|----------|--------|------|--------|-------|
| **Primary Star** | Largest circle | Spectral-class colour | White 1px | `Class+Grade` (e.g. `G2`) |
| **Companion Star** | Medium circle | Spectral-class colour | White 1px | `Class+Grade` |
| **Circumstellar Disk** | Scattered point ring (backlog) | Brown/grey `#8B7355` | Same | `Disk` |
| **Dwarf Planet** | Small circle | Grey `#9CA3AF` | Dark grey | `Dwarf` |
| **Terrestrial World** | Small circle | Green/brown `#4ADE80` / `#A16207` | Dark green | `Terrestrial` |
| **Ice World** | Small circle | Cyan/white `#22D3EE` | Dark cyan | `Ice` |
| **Gas World Class I** | Medium circle | Pale yellow/banded `#FDE047` | Dark yellow | `Gas I` |
| **Gas World Class II** | Medium circle | Blue banded `#60A5FA` | Dark blue | `Gas II` |
| **Gas World Class III** | Medium circle | Red/orange `#FB923C` | Dark orange | `Gas III` |
| **Gas World Class IV** | Medium circle | Violet `#C084FC` | Dark violet | `Gas IV` |
| **Gas World Class V** | Medium circle | White/grey `#E5E7EB` | Dark grey | `Gas V` |
| **Main World** | Highlighted circle | Same as its type | **Gold 2px** `#FACC15` | `★ MAIN` + type |

**Spectral Class Colours:**
- O: `#A5C8FF` (blue-white)
- B: `#C2D8FF`
- A: `#FFFFFF`
- F: `#FFF8E7`
- G: `#FFE4B5` (yellow)
- K: `#FFB366` (orange)
- M: `#FF6B6B` (red)

### 5.2 Orbit Lines

- Thin concentric rings or elliptical arcs.
- Colour: `#FFFFFF` at 15% opacity.
- Main world orbit: slightly thicker, 30% opacity.
- Hot Jupiter cleared zones: faint radial gradient shadow behind the gas giant's orbit.

### 5.3 Scale & Layout

- **Logarithmic radial scaling** so 0.1 AU and 80 AU bodies are both visible.
- Star sits at the visual centre `(0,0)`.
- Companion stars: placed at their `orbitDistance` on a fixed radial line (simplified; true barycentric orbits are backlog).
- **No true-scale sizing** — sizes are mass-tier abstractions so labels remain readable.

### 5.4 What Is NOT in MVP (Backlog)

- **Rings** — no planetary ring rendering rules exist in MWG yet.
- **INTRAS Level 2** — no moons, no child bodies orbiting planets. Only INTRAS Level 1 (stars, disks, planets).
- **True barycentric orbits** — companion stars are static/fixed-angle for now.
- **Brachistochrone trajectory lines** — UI exists in backlog (FR-031).

### 5.5 Disk Rendering Backlog Specification
**Status:** Backlog — to be implemented in Phase 5+ (Polish).

**Requirement:** Circumstellar disks must **not** be rendered as simple dashed rings in the final visualiser. Instead, each disk should be drawn as a **scattered high-density point field** (noise) arranged along its orbital ring.

**Visual specification:**
- Each disk consists of **300–800 tiny points** (0.5–1.5 px radius) distributed along its orbital circumference.
- Point density is **radially constrained** to a narrow band (e.g., ±3–5% of the orbital radius) so it reads as a disk rather than a smear.
- Angular placement is **randomly jittered** around the full 360°.
- Opacity varies per point (20%–70%) to create texture and depth.
- Multiple disks in the same system must be visually distinct from one another and from the background starfield. This is achieved by:
  - Using a slightly warmer/browner colour palette than background stars (`#8B7355`, `#A0522D`, `#CD853F`)
  - Higher point density along a narrow orbital track
  - Larger average point size than background stars
- Disk points are **static** (do not orbit individually) but rotate with the disk's orbital period as a whole if animated.

**Rationale:** A dashed ring looks like a wireframe diagram. A scattered point ring looks like an actual debris/asteroid belt and provides the gritty, realistic aesthetic appropriate for a referee aid.

**Seeding:** Disk point distributions are derived from the `starfieldSeed` + disk index so the same system always renders the same disk shape.

## 6. The Procedural Starfield Background

### 6.1 Requirement

The background must **not** use JPG or PNG assets. It is a **procedural vector starfield** generated at runtime from a seed string.

### 6.2 The Seed System

- Every time a map is opened, MWG generates a random **8-character alphanumeric seed** (e.g. `X7Kp9mQ2`).
- This seed is part of the map payload.
- The 2D map feeds the seed into a **deterministic PRNG** (e.g. Mulberry32 or LCG) to place stars.
- **Same seed = same starfield**, everywhere.

### 6.3 User Controls

In the 2D map UI, display:
- **Seed:** `X7Kp9mQ2`
- **Regenerate** button (rolls a new random seed, redraws background).
- **Copy** button (copies seed to clipboard).
- **Paste** input (accepts another player's seed, redraws background).

Changing the seed does **not** change the planetary system; it only changes the wallpaper.

### 6.4 Visual Effect: Random Stellar Noise

The background is drawn on a full-screen Canvas layer **behind** the orbits.

Each "star" in the noise field is:
- A tiny circle (0.5–2 px radius).
- Slightly varying opacity (10%–80%).
- Slightly varying colour (white, pale blue, pale yellow).
- Density: ~200–400 stars across a 1080p viewport, regenerated on canvas resize.

**Optional enhancement:** a very faint nebula cloud using low-opacity radial gradients, also seeded.

## 7. Animation & Time Control

### 7.1 Epoch

- **Default epoch:** `2300-01-01` CE.
- All initial orbital angles are **randomly determined** for this epoch when the map is first opened.
- The epoch and angles are derived from the `StarSystem` data (optionally hashed from the system ID for reproducibility).

### 7.2 Animation Loop

- **Single `requestAnimationFrame` loop** updates every body each frame.
- No per-object timers.
- Loop runs continuously but can be paused.

### 7.3 Time Controls

| Control | Behaviour |
|---------|-----------|
| **Play / Pause** | Toggles the animation loop. |
| **Speed** | Slider: `0.25x`, `0.5x`, `1x` (default), `2x`, `5x`, `10x`, `30x`, `365x`. |
| **Reverse** | Toggle to make time run backward. |
| **Step +1 Day** | Advances exactly one day, pauses if playing. |
| **Step -1 Day** | Rewinds exactly one day. |
| **Step +7 Days** | Advances one week. |
| **Step -7 Days** | Rewinds one week. |
| **Reset** | Snaps back to `2300-01-01`. |

### 7.4 Orbital Motion Math (MVP Simplified)

For each body:
- Orbital period `T` is derived from `distanceAU` using Kepler's 3rd Law: `T = k × a^(3/2)`.
- Given a simulation day offset `Δt`, the body's angle advances by `2π × Δt / T`.
- Direction is **prograde** (counter-clockwise) for all bodies in MVP.
- **Realistic directional variance** (some bodies retrograde) is backlog.

The body's screen position:
```
x = cx + cos(angle) × scaledRadius
y = cy + sin(angle) × scaledRadius
```

Where `scaledRadius` comes from the logarithmic distance mapping.

## 11. Glossary

| Term | Meaning |
|------|---------|
| **INTRAS Level 1** | Intrastellar (primary) elements: stars, circumstellar disks, and planets that orbit the primary star directly. |
| **INTRAS Level 2** | Intrastellar child elements: moons and ring systems orbiting a planet. **Backlog.** |
| **The Path** | The data bridge from MWG to the 2D map: a Base64-encoded query-string payload. |
| **Starfield Seed** | An 8-character alphanumeric string that deterministically generates the background noise field. |
| **Epoch** | The simulated calendar date from which all orbital positions are calculated. Default: `2300-01-01`. |
