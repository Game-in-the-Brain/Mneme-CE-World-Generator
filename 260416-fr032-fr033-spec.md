## Executive Summary

This document unifies the expanded **FR-032** (Economic Engine Customization) and **FR-033** (Goal-Loop Generation) requirements. It replaces the older Boat-Unit-only FR-032 spec with a full **world-building economic toolkit** that lets users choose between CE/Traveller stagnation and Mneme compounding growth, then calibrate the tables that drive generation.

## Part 1 — FR-032: Economic Engine Customization

### Philosophy: Why This Exists

**CE/Traveller** assumes income and productivity stagnate after a certain Tech Level. TL becomes merely a measure of jump distance, not human progress. This creates a world without growth, where "every job opportunity doesn't feel real" because there are no market forces — no demand and supply.

**Mneme** rejects this. Stagnation is a form of caste system. High growth is opportunity. The entire purpose of Mneme World Generation is to populate a realistic setting where planets are visited because they are *used* — for terraforming, for building the terraton structures (giant serpents, space elevators, solar swarms) that make interstellar travel possible. The economy demands more Earth-like worlds and more worlds to inhabit.

However, many users come from a CE/Traveller mental model. The solution is **not** to force one answer. It is to expose the underlying tables and let the user (or referee) choose their economic assumptions.

### 1.1 TL Productivity Settings Panel (Settings Page)

**Location:** New section in `Settings.tsx` titled **"Economic Assumptions"**

#### Preset Selector
- **Dropdown:** `Mneme` (default) | `CE / Traveller` | `Custom`
- **Actions beside dropdown:**
  - **Save** — persists current custom values to `localStorage`
  - **Save As** — prompts for a name, saves a new preset
  - **Load** — loads a previously saved custom preset
  - **Import** — loads a JSON file containing a preset
  - **Export** — downloads current values as JSON

#### Preset Data Structure
```typescript
interface TLProductivityPreset {
  id: string;
  name: string;
  description: string;

  // Direct override table (only used when curve === 'custom')
  soc7IncomeByTL?: Record<number, number>;
}
```

### 1.4 Development Table Customization

**UI Component:** Second collapsible card in Settings — **"Development Distribution"**

**Current Mneme Default (from `getDevelopment` in `worldData.ts`):**

| 2D6 | Level | HDI | SOC |
|---|---|---|---|
| 2 | UnderDeveloped | 0.0–0.39 | 2 |
| 3–5 | UnderDeveloped | 0.40–0.49 | 3 |
| 6–7 | UnderDeveloped | 0.50–0.59 | 4 |
| 8 | Developing | 0.60–0.69 | 5 |
| 9 | Mature | 0.70–0.79 | 6 |
| 10 | Developed | 0.80–0.89 | 8 |
| 11 | Well Developed | 0.90–0.94 | 9 |
| 12 | Very Developed | >0.95 | 10 |

**Customization:**
- User can **toggle the probability weight** of each 2D6 result (2–12).
- Default weight = 1 for all results (uniform 2D6 distribution).
- A weight of 0 removes that result from the pool.
- Weights are normalized into probabilities internally.
- **Preset buttons:** `Mneme Default` | `CE Optimistic` (reduces UnderDeveloped weight) | `Reset Uniform`

**Example:** A user who wants a grittier setting could set rolls 2–7 to weight 2 and rolls 9–12 to weight 0.5, making underdeveloped worlds more common.

**Storage key:** `mneme_generator_options.developmentWeights`

**Generation impact:** `getDevelopment()` will accept an optional `weights` override. If none provided, uses uniform (backward compatible). If a preset is active, uses weighted random selection.

### 1.5 Engine Integration — WB Drives the Underlying Economy

**This is the critical architectural requirement:** The World Builder is not a display layer. When the user changes the productivity curve or the Boat anchor, the **entire economic engine recalculates**.

#### The Boat Anchor
Instead of editing raw SOC 7 incomes directly, the primary calibration input is:

> **"Years for a SOC 7 worker to purchase the base Boat (10DT)"**

- Boat price is fixed from `mneme_ship_reference.json` at **5,320,400 Cr**.
- `SOC 7 annual income = Boat Price / Y`
- `gdpPerDay = SOC 7 annual income / 365`

**Example:**
- **Mneme default:** Y = 30 yrs → SOC 7 annual = 177,347 Cr/year → ~486 Cr/day at TL 7
- **CE stagnant:** Y ≈ 222 yrs → SOC 7 annual = 24,000 Cr/year → ~66 Cr/day at **all TLs**

This `gdpPerDay` becomes the economic baseline. Higher TLs scale according to the selected curve:
- **Mneme:** compounding growth from `GDP_PER_DAY_BY_TL`
- **CE:** flat line — 2,000 Cr/month SOC 7 at every TL
- **Linear:** fixed percentage increase per TL step

The resulting table **replaces** the hardcoded `GDP_PER_DAY_BY_TL` in `worldData.ts` for the current session.

#### Trade Values & Ships
`calculateStarport()` uses the preset's dynamic `gdpPerDay`:

```
annualTrade = population × gdpPerDay × 365 × tradeFraction × wealthMultiplier
```

Because `gdpPerDay` drops drastically in CE mode:
- `annualTrade` drops
- `weeklyBase` drops
- `weeklyActivity` drops
- `shipsInArea.ts` budget drops
- **Far fewer ships appear** at the port

Ship prices in `mneme_ship_reference.json` remain fixed in credits, but they are now enormously more expensive in human terms. A 44 M Cr Courier is 8× the Boat price — in CE mode that is **20,000 income-years** instead of **240 income-years**.

#### Fallback Rule
If no preset is loaded, the generator uses the **Mneme default** (backward compatible).

> **CE/Traveller stagnates.** In classic rules, Tech Level only measures jump distance. Income and productivity flatline. There are no market forces because there is no growth — every "job opportunity" is static, and the vast developing populations of our real world (Africa, South Asia, Southeast Asia, South America, Middle East) simply don't exist in the setting. Everything is gentrified.

A small colony with high automation can maintain significant starport traffic because robots don't sleep.
>

### 1.9 World Economics Mechanics Reference

This section explains the exact math that runs when the World Builder changes productivity assumptions. It belongs inline in the Settings UI as a collapsible **"How does this work?"** panel.

#### The Four-Step Pipeline

**Step 1 — From Preset to Per-Capita Income**
```
SOC 7 annual income = Boat Price / boatYears
                    = 5,320,400 / Y

SOC 7 monthly income = SOC 7 annual / 12
gdpPerDay = SOC 7 annual / 365
```

For CE mode (`curve = 'flat'`):
- This `gdpPerDay` is applied **identically** to every TL (7–16).

For Mneme mode (`curve = 'mneme'`):
- The TL 7 value is anchored by `boatYears`.
- TL 8–16 use the existing Mneme compounding curve:
  - `gdpPerDay[tl] = gdpPerDay[7] × (mnemeCurve[tl] / mnemeCurve[7])`

**Step 2 — From Population to GDP**
```
GDP/year = Population × gdpPerDay × 365
```

**Step 3 — From GDP to Port Trade**
```
Annual Port Trade = GDP/year × TradeFraction(development) × WealthMultiplier(wealth)
```

Trade Fraction (how much of the economy flows through the port):
- UnderDeveloped: 5%
- Developing: 10%
- Mature: 15%
- Developed: 20%
- Well Developed: 25%
- Very Developed: 30%

Wealth Multiplier:
- Average: ×1.0
- Better-off: ×1.2
- Prosperous: ×1.5
- Affluent: ×2.0

**Step 4 — From Port Trade to Starport Class & Ships**
```
PSS = floor( log10(Annual Port Trade) ) − 10

Raw Class = f(PSS):
  PSS < 4  → X
  4–5      → E
  6–7      → D
  8–9      → C
  10–11    → B
  ≥12      → A

Final Class = min(Raw Class, TL Cap)

Weekly Base = Annual Port Trade ÷ 364
Weekly Activity = Weekly Base × 3D6 (range 3–18)
```

The `Weekly Activity` becomes the **budget** for `shipsInArea.ts`. A smaller `gdpPerDay` directly shrinks this budget, which reduces the number and size of ships generated.

#### Why This Matters in Practice

| Mode | TL 9 SOC 7 Monthly | 400K-pop Annual Trade | Weekly Activity (avg) | Typical Ships |
|---|---|---|---|---|
| **Mneme** | 44,580 Cr | ~54 B Cr | ~1.5 B Cr | Dozens (including liners) |
| **CE Flat** | 2,000 Cr | ~2.4 B Cr | ~66 M Cr | A handful of small craft |

*Same population. Same world. Different economic assumptions. Different stories.*

## Part 2 — FR-033: Goal-Loop Generation

### Objective

On the **Generator Dashboard**, let the user set target constraints and auto-generate systems until a match is found. This is especially useful for referees who need "a TL 12+ world with Class A starport and Hab > 0" for a specific adventure hook.

### UI Addition

A new collapsible card below the generation controls in `GeneratorDashboard.tsx`:

```
┌─ Targeted Generation ──────────────────────────────────────────┐
│                                                                │
│  Starport Class:  [ Any  ▼ ]  (X, E, D, C, B, A)              │
│  Min Population:  [ Any  ▼ ]  (Any, 1K, 10K, 100K, 1M, 1B)    │
│  Min Habitability: [ Any  ▼ ]  (Any, ≥0, ≥1, ≥2, ≥3)          │
│                                                                │
│  [ □ ] Require populated world                                │
│  [ □ ] Stop after first match                                 │
│                                                                │
│  [ Generate Until Match ]                                     │
│                                                                │
│  Last run: 47 iterations — Match found!                        │
└────────────────────────────────────────────────────────────────┘
```

### Behavior

1. User clicks **Generate Until Match**.
2. A `while` loop runs `generateStarSystem(options)` up to a **safety limit** of **1,000 iterations**.
3. After each generation, the system is checked against active constraints.
4. **First match** is returned and displayed.
5. The loop counter is shown to the user.
6. If the safety limit is hit without a match, show: *"No match found in 1,000 attempts. Try relaxing your constraints."*

### Match Logic

```typescript
function matchesTarget(system: StarSystem, targets: GenerationTargets): boolean {
  if (targets.starportClass && system.inhabitants.starport.class !== targets.starportClass) return false;
  if (targets.minPopulation && system.inhabitants.population < targets.minPopulation) return false;
  if (targets.minHabitability !== undefined && system.mainWorld.habitability < targets.minHabitability) return false;
  if (targets.populatedOnly && system.inhabitants.populated === false) return false;
  return true;
}
```

### Performance Guardrails

- Loop is synchronous but capped at 1,000.
- UI shows a spinner: *"Searching… iteration 47"*.
- If `stopAfterFirstMatch` is unchecked (advanced), return the *best* match (closest to targets) instead of first — future enhancement; Phase 1 only implements first-match.
